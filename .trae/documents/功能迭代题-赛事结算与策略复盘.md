# 功能迭代题：赛事结算与策略复盘系统

## 出题元信息

- 项目编号：253
- 项目类型：Phaser.js + React + TypeScript Web 游戏
- 任务类型：Feature 迭代
- 业务领域：游戏开发（含 UI/渲染） + Web 前端
- 推荐难度：偏高
- 修改范围：跨模块多文件
- 预期涉及模块：`App.tsx`、`RaceView`、`Results`、`HighScores`、`useGameStore`、`useUISTore`、`RaceScene`、`storage`、类型定义与工具函数
- 评测重点：状态流理解、跨组件数据贯通、游戏事件归因、结果页可视化、历史记录兼容性、指令遵循

## 题目 Prompt

请基于当前 253 项目继续迭代 “F1 赛车进站策略与轮胎管理挑战” 小游戏，实现一个完整的 **赛事结算与策略复盘系统**。

当前项目已经具备基础比赛、进站、轮胎、燃油、天气、安全车、undercut、最高分记录等功能，但比赛结束后的结算体验和策略复盘能力不足。你需要在不推翻现有架构的前提下，补齐从比赛结束到复盘、再到本地记录保存的完整闭环。

### 需求一：完赛结算页必须稳定可达

比赛达到总圈数或玩家因爆胎/燃油耗尽退赛时，必须进入结算页，而不是回到主菜单或停留在比赛画面。

结算页至少展示：

- 最终排名、总用时、最佳圈速、完成圈数、退赛状态
- 总得分与策略评分
- 使用过的轮胎序列
- 玩家进站记录，包括圈数、换胎类型、加油量、损失时间
- 比赛过程中发生过的关键事件摘要，包括降雨、安全车、事故、undercut
- 至少 3 条基于玩家表现生成的策略建议

### 需求二：新增策略复盘时间线

在结算页新增一个 “策略复盘时间线” 区域，用时间线形式展示比赛关键节点。

时间线节点至少包括：

- 开赛初始轮胎和燃油
- 每次玩家进站
- 天气变化事件
- 安全车出动与结束
- 对手 undercut
- 玩家爆胎风险、燃油警告或退赛原因
- 完赛或退赛节点

每个节点需要包含圈数、标题、说明、严重程度或类型标识，并使用现有赛车主题视觉风格展示。

### 需求三：扩展本地最高分记录

最高分记录不再只保存基础分数，还需要保存本场比赛的复盘摘要。

历史记录页面需要支持展开单条记录，查看：

- 轮胎使用序列
- 进站次数与每次进站摘要
- 关键事件数量
- 策略建议摘要
- 是否退赛及退赛原因

已有旧版 localStorage 记录必须兼容，不能因为字段缺失导致最高分页面白屏或报错。

### 需求四：学习模式复用复盘能力

策略学习模式完成比赛后，结算页需要额外显示 “学习反馈” 区域。

学习反馈至少包含：

- 玩家是否及时响应降雨
- 是否在轮胎严重磨损前安排进站
- 燃油负载是否过重或过轻
- 是否错过安全车进站窗口

大奖赛模式不显示学习反馈区，但仍显示普通策略建议。

### 需求五：交互与 UI 要求

- 结算页不能是纯文本堆叠，需要保持当前项目的 F1 仪表盘、玻璃态面板、轮胎色标、事件警报风格。
- 时间线和历史记录展开区域需要在 1280x720 桌面分辨率下可读，不允许关键内容被遮挡。
- 历史记录展开/收起必须是本地交互，不允许刷新页面。
- 比赛结束后再次点击 “再来一局” 必须能重新进入比赛，并且上一局复盘数据不能污染新比赛。
- 返回主菜单后再次查看最高分，刚刚的复盘摘要仍能看到。

## 明确约束

### 技术栈或依赖约束

- 必须继续使用 React + TypeScript + Phaser.js。
- 禁止新增第三方 npm 依赖。
- 禁止引入后端服务，所有记录继续使用浏览器 localStorage。

### 架构或模式约束

- 不允许把所有逻辑堆到单个组件里。
- 比赛事件采集、复盘数据构建、结果页展示、本地存储兼容处理需要拆分到合适的模块或工具函数中。
- 不允许重写 Phaser 主循环；只能在现有 `RaceScene`、store、系统类基础上增量扩展。

### 代码风格或规范约束

- 新增 TypeScript 类型必须显式声明，禁止使用大面积 `any` 逃避类型设计。
- 新增 UI 文案必须使用中文，且不得出现乱码。
- 新增复杂逻辑需要保留简短注释说明意图。

### 业务逻辑约束

- 同一场比赛只能保存一次最高分与复盘记录，不能重复写入 localStorage。
- 旧历史记录字段缺失时必须降级显示，不允许崩溃。
- 退赛也必须进入结算页，但分数和建议需要体现退赛惩罚。
- 复盘建议必须根据真实比赛数据生成，不能固定写死三条模板。

### 非代码回复约束

- 完成后请用 Markdown 输出：
  - 修改的核心模块列表
  - 复盘数据结构说明
  - 本地验证命令
  - 你覆盖的关键用户路径

## 验收标准

### 基础验收

- `npm run build` 或项目等价构建命令通过。
- 普通大奖赛完赛后能进入结算页。
- 玩家退赛后能进入结算页并显示退赛原因。
- 结算页能显示时间线、策略建议、进站记录和关键事件摘要。
- 学习模式结算页额外显示学习反馈。
- 最高分页面能展开查看复盘摘要。
- 旧版 localStorage 数据不会导致最高分页面报错。

### 交互验收

- 从主菜单进入大奖赛，完成比赛后进入结算页。
- 从主菜单进入学习模式，完成教程和比赛后进入带学习反馈的结算页。
- 在比赛中触发进站后，结算页能看到对应进站节点。
- 在比赛中触发安全车或降雨后，结算页能看到对应事件节点。
- 结算页点击 “再来一局” 后，新比赛数据从零开始。
- 结算页点击 “返回主菜单” 后，再进入最高分记录，能查看刚才保存的复盘摘要。

## 建议评分关注点

- 是否真正理解当前 `raceState` 与 `currentView` 的分工，避免结算页再次被错误卸载。
- 是否能设计可扩展的复盘数据结构，而不是临时拼字符串。
- 是否处理了旧 localStorage 数据兼容。
- 是否避免重复保存成绩。
- 是否保持现有 UI 风格，而不是生成突兀的新页面。
- 是否能把 Phaser 事件、Zustand store、React 展示层串起来。
- 是否有真实的数据驱动策略建议，而不是固定文案。

## 预期难点

- 当前项目的比赛结束逻辑横跨 `RaceScene`、`useGameStore`、`RaceView`、`App`，需要理清视图切换和状态更新顺序。
- 本地最高分记录已有旧结构，扩展字段时需要兼容历史数据。
- 复盘时间线需要从多个来源采集数据：进站结果、随机事件、天气状态、警告通知、退赛原因。
- 学习模式和大奖赛模式需要共享结算页主体，但显示不同反馈区域。
- 重新开始比赛时必须清理上一局复盘状态，避免数据串场。
